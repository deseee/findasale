# Health Report — 2026-03-22

## Summary
Overall codebase health is **Yellow**. One critical unauthenticated admin endpoint was found that requires immediate attention before the next user-facing deploy: the organizer verification approve/reject routes have no auth or admin guard — any anonymous caller can approve or reject any organizer's verification status. A secondary high-severity finding is a JWT signing fallback to a known public string (`'your_secret_key'`) in one passkey controller path. The `/api/dev` route remains accessible in production without a NODE_ENV guard (mitigated by auth + email gate, but should still be hardened). Frontend and SSR scans came back entirely clean. Code quality is very good with zero console.log noise in pages and no unhandled promise rejections found.

## Critical (blocker — fix before next deploy)

### C1 — Unauthenticated Admin Verification Routes
**File:** `packages/backend/src/routes/verification.ts` lines 21–25
**Routes:**
```
POST /api/verification/admin/:organizerId/approve
POST /api/verification/admin/:organizerId/reject
```
**Finding:** Neither route has `authenticate` or `requireAdmin` middleware. The controller (`verificationController.ts:96` and `:136`) accepts `req: Request` (not `AuthRequest`) and performs no identity check. Any unauthenticated caller who knows an `organizerId` can approve or reject verification for any organizer.
**Fix:** Add `authenticate, requireAdmin` middleware to both routes:
```ts
router.post('/admin/:organizerId/approve', authenticate, requireAdmin, adminApproveVerification);
router.post('/admin/:organizerId/reject', authenticate, requireAdmin, adminRejectVerification);
```
**Severity:** P0 — data integrity attack vector; trivially exploitable.

## High (fix this week)

### H1 — JWT Signing Fallback to Known Public Value
**File:** `packages/backend/src/controllers/passkeyController.ts` line 373
**Finding:**
```ts
process.env.JWT_SECRET || 'your_secret_key',
```
If `JWT_SECRET` is not set in production, tokens are signed with the string `'your_secret_key'` — a value publicly visible in `.env.example`. The rest of the codebase uses `process.env.JWT_SECRET!` (non-null assertion that throws if unset), making this inconsistent. In Railway/production this is mitigated if `JWT_SECRET` is properly set, but the fallback is a silent failure mode.
**Fix:** Replace with a hard throw like the rest of the codebase:
```ts
const jwtSecret = process.env.JWT_SECRET;
if (!jwtSecret) throw new Error('JWT_SECRET is not set');
// ... use jwtSecret
```
**Severity:** P1 — token forgery risk if `JWT_SECRET` ever goes unset in a new environment or rotation.

### H2 — `/api/dev` Route Accessible in Production Without NODE_ENV Guard
**File:** `packages/backend/src/index.ts` line 376
**Finding:**
```ts
app.use('/api/dev', devRoutes); // Dev utilities
```
No `if (process.env.NODE_ENV !== 'production')` guard. The route does require `authenticate` middleware and the `fix-seed-tiers` endpoint has an email gate (`user1@example.com` only), limiting actual blast radius. However, dev utility endpoints should not be reachable in production at all.
**Fix:** Wrap the mount in a NODE_ENV check:
```ts
if (process.env.NODE_ENV !== 'production') {
  app.use('/api/dev', devRoutes);
}
```
**Severity:** P1 — defense-in-depth gap; remove attack surface.

### H3 — Missing `MAILERLITE_API_KEY` in Backend `.env`
**File:** `packages/backend/.env` (env config mismatch)
**Finding:** `MAILERLITE_API_KEY` is in `.env.example` and actively used by `mailerliteService.ts` for shopper subscriber enrollment (on registration) and sale publishing events. The service degrades gracefully with a `console.warn`, but **email automation is silently disabled in local dev**, meaning onboarding flows are untestable locally without this key.
**Fix:** Add the MailerLite API key to `packages/backend/.env`. Obtain from MailerLite → Integrations → API.
**Severity:** P1 — silent feature degradation; local dev misrepresents production email behavior.

## Medium (fix this sprint)

### M1 — Unbounded `findMany` in Export Controller
**File:** `packages/backend/src/controllers/exportController.ts` lines 400, 417, 437
**Finding:** Three `prisma.findMany()` calls (sales, items, purchases) in the CSV export path have no `take` limit. An organizer with thousands of items triggering a full CSV export could cause memory spikes or timeout.
**Fix:** Add `take` limits with pagination or a hard cap (e.g., `take: 5000`), or implement streaming export.
**Severity:** P2 — performance risk at scale; not an immediate concern during beta with few organizers.

### M2 — 13 TODO/FIXME/HACK Markers in Codebase
**Scope:** Distributed across backend controllers and services
**Finding:** 13 `TODO`, `FIXME`, or `HACK` markers found. None are in security-critical paths based on spot checks.
**Fix:** Audit and convert each to a tracked issue or remove if resolved.
**Severity:** P2 — code hygiene; no immediate risk.

### M3 — Missing DEFAULT_* Region Vars in Backend `.env`
**File:** `packages/backend/.env`
**Finding:** `DEFAULT_CITY`, `DEFAULT_STATE`, `DEFAULT_STATE_ABBREV`, `DEFAULT_LAT`, `DEFAULT_LNG`, `DEFAULT_RADIUS_MILES`, `DEFAULT_TIMEZONE` are in `.env.example` but not in `.env`. `regionConfig.ts` has hardcoded fallback values (Grand Rapids, MI coords), so no runtime failure occurs — but local dev diverges from the `.env.example` contract.
**Fix:** Add these to `packages/backend/.env` with the Grand Rapids values.
**Severity:** P2 — config drift; no immediate impact.

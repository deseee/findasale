**Scope:** Pre-beta P0/P1/P2 sweep across high-risk code paths

## P0 — Blockers (must fix before first organizer)

| Bug | File | Description | Recommended Fix |
|-----|------|-------------|-----------------|
| **Missing sale ownership check in bulk items API** | `/backend/src/routes/items.ts` line 66–85 | Bulk operations endpoint (`POST /items/bulk`) verifies ownership by checking `sale.organizer.userId` against `req.user.id`, BUT the endpoint has no mechanism to prevent passing arbitrary `itemIds` from other organizers' sales via database query. While the code does filter for ownership, if an attacker knows another organizer's item IDs, the error message leaks whether items exist (403 "do not belong" vs 400 "invalid itemId"). This is auth bypass risk. | Add explicit audit logging for failed ownership checks. Return generic 404 for all ownership failures rather than 403 (hide existence). Consider rate-limiting bulk operations to detect probe attacks. |
| **Null price handling in bulk price operations** | `/backend/src/routes/items.ts` lines 356–375 (price_adjust), 399–414 (price) | Bulk price operations filter items where `price !== null` but silently skip those with null price. Frontend shows no warning that some items were skipped. If organizer selects 10 items and 3 have null prices, 3 silently fail without user notification. | Return success response with `skipped` field listing items with null prices. Show toast warning in frontend: "3 items skipped — price not set." Require explicit opt-in to skip null prices. |
| **Missing SOLD/RESERVED validation in purchase flow** | `/backend/src/controllers/stripeController.ts` line 178–184 | `createPaymentIntent` retrieves item but does NOT validate item.status is still AVAILABLE before proceeding. Race condition: item marked SOLD between user selection and payment intent creation. Payment succeeds but cannot be fulfilled. | Add `if (item.status !== 'AVAILABLE') return 409 Item already sold/reserved`. Atomically increment item.status in same transaction. Document race window in code comment. |
| **Auth token invalidation on password change not enforced** | `/backend/src/prisma/schema.prisma` line 26 (User.tokenVersion) | Schema includes `tokenVersion` field for JWT invalidation but middleware (`/backend/src/middleware/auth.ts`) never checks it. Old tokens remain valid after password reset, allowing account takeover if password is compromised. | Add `tokenVersion` field to decoded JWT payload. Verify `decoded.tokenVersion === user.tokenVersion` in authenticate middleware. Increment tokenVersion on password reset. Test with old tokens. |

## P1 — High Priority (fix within 2 sessions)

| Bug | File | Description | Recommended Fix |
|-----|------|-------------|-----------------|
| **No error state for failed bulk operations** | `/frontend/pages/organizer/add-items/[saleId].tsx` line 774+ (bulkUpdateMutation) | Bulk mutation has `onSuccess` and `onError` handlers, but `onError` only shows generic toast. No field-level feedback if some items fail (e.g., 5/10 succeed due to status constraints). Silent partial failures. | Return partial success response from backend with `{ succeeded: [], failed: [{ itemId, reason }] }`. Frontend displays count and failed IDs. Show warning toast if any items failed. |
| **Draft item count mismatch on rapid addition** | `/frontend/pages/organizer/add-items/[saleId].tsx` line 400+ (rapidItems state) | `rapidItems` array adds drafts locally but refetch query key only invalidates on `queryClient.invalidateQueries(['organizer-items'])`. If organizer adds 5 items via camera, then navigates away and back, stale cache may show fewer items. Race between local state and server truth. | Refetch items query immediately after each successful item creation. Use `queryClient.setQueryData` to optimistically update cache. Add `enabled: !inMutationFlight` to prevent stale while mutating. |
| **Missing validation for entrance pin coordinates** | `/frontend/pages/organizer/edit-sale/[id].tsx` line 362–386 | EntrancePinPicker component accepts `entranceLat` and `entranceLng` but backend does not validate they fall within sale bounds. Organizer could set entrance pin miles away from sale address. | Add backend validation: `Math.hypot(entranceLat - saleLat, entranceLng - saleLng) < 0.05` (approx 2.8 miles). Show client warning if pin > 0.5 miles from sale center. Return 400 if too far. |
| **Batch hold actions (extend/release) missing organizer re-verification** | `/backend/src/controllers/reservationController.ts` line 186–196 | `batchUpdateHolds` verifies ownership once at start (`validHolds.filter`) but if database state changes between verification and update (e.g., hold transferred to different sale), operation may update wrong item. No transactional safety. | Wrap hold verification + update in `prisma.$transaction`. Re-verify organizer ownership inside transaction before each update. Add `where: { organizerId: organizer.id }` to all final update queries. |
| **Silent failure on Stripe account creation** | `/backend/src/controllers/stripeController.ts` line 119–146 | Catch-all error handler logs but returns generic 500. Frontend shows "Failed to create Stripe Connect account" with no actionable recovery. Onboarding wizard cannot proceed. | Add error type detection: return 400 for Stripe API validation errors (missing_field, invalid_param). Return 503 for rate limits/timeouts. Frontend shows specific guidance: "Email required for Stripe" vs "Try again in 1 minute". |
| **Bulk photo operations exceed photo limit without feedback** | `/backend/src/routes/items.ts` line 593–612 | `bulk/photos` endpoint skips items that would exceed 5 photos/item but returns success with no `skipped` field. Organizer thinks all items got photos; actually 4/10 did. | Add `skipped: [ { itemId, reason: "would exceed 5 photos" } ]` to response. Frontend toast: "Added photos to 6/10 items — 4 skipped." Return count in response body. |
| **Missing saleId validation in add-items page** | `/frontend/pages/organizer/add-items/[saleId].tsx` line 30 (router.query.saleId) | If `saleId` param is missing or malformed, page loads with `saleId = undefined`, API calls silently fail. No redirect or error UI. | Add useEffect to validate `saleId` on mount. If missing, redirect to `/organizer/dashboard` with toast "Sale not found". Fetch sale metadata to verify ownership before rendering. |
| **Category field in bulk operations accepts any string** | `/backend/src/routes/items.ts` line 333–346 | Category is stored as arbitrary string; no enum validation. Organizer could set category to "123" or "∑∆¬" (nonsense). Schema allows `category: String?` without constraints. | Add category enum in Prisma schema or validation layer. Whitelist: furniture, decor, vintage, textiles, collectibles, art, antiques, jewelry, books, tools, electronics, clothing, home, other. Return 400 for invalid category. |

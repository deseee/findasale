# Health Report — 2026-03-15

## Summary

Overall codebase health is **Yellow**. Auth coverage is solid — every mutating route correctly uses `router.use(authenticate)` as a catch-all and no CORS wildcards or JWT bypass issues were found. The most actionable finding is a **silent env var mismatch for MailerLite**: the service reads `MAILERLITE_API_KEY` but `.env` defines `MAILERLITE_API_TOKEN`, meaning all email-marketing API calls have been silently failing with no error — only a `console.warn`. Several DEFAULT_* geocoding env vars are also missing from `.env`, which may affect default-region fallback behavior. Accessibility and SSR scans came back clean.

## High (fix this week)

### H1 — MailerLite API Key Name Mismatch (Silent Failure)
**File:** `packages/backend/src/services/mailerliteService.ts:15`
**Issue:** Code reads `process.env.MAILERLITE_API_KEY`, but `.env` defines `MAILERLITE_API_TOKEN`. The service degrades silently (emits `console.warn` and returns early). All MailerLite subscriber sync calls (sale_published field updates, shopper group subscriptions) are currently no-ops.
**Fix:** Rename `MAILERLITE_API_TOKEN` → `MAILERLITE_API_KEY` in `.env`, OR rename the env var read in `mailerliteService.ts` to match the actual key name.

## Medium (fix this sprint)

### M1 — Missing DEFAULT_* Env Vars
**Issue:** `.env.example` defines 8 default-region variables not present in `.env`:
`DEFAULT_CITY`, `DEFAULT_COUNTY`, `DEFAULT_LAT`, `DEFAULT_LNG`, `DEFAULT_RADIUS_MILES`, `DEFAULT_STATE`, `DEFAULT_STATE_ABBREV`, `DEFAULT_TIMEZONE`
**Risk:** Any code path that relies on these as fallback region values may behave unpredictably (undefined values, incorrect default search coordinates).
**Fix:** Add these vars to `packages/backend/.env` with appropriate values for the primary launch market.

### M2 — Planner AI Endpoint Unauthenticated (AI Cost Exposure)
**File:** `packages/backend/src/routes/planner.ts:20`
**Issue:** `POST /api/planner/chat` is intentionally public (no auth) with only a rate limiter (30 req/15min per IP). IP-based rate limiting is bypassable via proxy rotation. Each call hits the AI inference layer.
**Risk:** Low-medium — moderate cost exposure, not a data leak. Acceptable for MVP but worth noting.
**Fix (optional):** Add optional auth to allow higher rate limits for authenticated users, stricter limits for guests. Or add a simple token-cost circuit breaker.

### M3 — Unbounded `findMany` in Export and Admin Controllers
**Files:**
- `packages/backend/src/controllers/exportController.ts:400,417,437` — `prisma.sale.findMany`, `prisma.item.findMany`, `prisma.purchase.findMany` without `take`
- `packages/backend/src/controllers/adminController.ts:83,205,271,283,294` — multiple `findMany` without `take`
**Risk:** Large datasets could cause memory pressure or slow responses. Admin endpoints are protected; export endpoints are auth-gated. Low immediate risk but could become a performance issue at scale.
**Fix:** Add pagination (`take`/`skip`) or a reasonable max cap (e.g., `take: 500`) to all export queries.

### M4 — `include: { organizer: true }` Without Select Scope
**File:** `packages/backend/src/controllers/authController.ts:106`
**Issue:** Referral lookup uses `include: { organizer: true }` which fetches the full organizer record. If the organizer model contains internal fields (tier, billing info, etc.) this is included in the Prisma object even if not returned to the client.
**Risk:** Low if response serialization strips it, but it's a hygiene issue. Auditing the `referrer` usage confirms it's only used internally (not serialized to response), so this is informational.
**Fix:** Narrow to `include: { organizer: { select: { id: true, userId: true } } }` for clarity.

## Medium (fix this sprint)

### M1 — `getItemsBySaleId` Has No Pagination Limit
**File:** `packages/backend/src/controllers/itemController.ts` line 324
**Route:** `GET /api/items?saleId=...` (public endpoint)
**Finding:** The primary endpoint for fetching all items in a sale runs `prisma.item.findMany()` with no `take` limit. All matching items are returned in a single query and response. For typical estate sales (50–200 items) this is acceptable today, but a large sale or adversarial request with a bogus saleId producing many results could strain memory and response time.
**Fix:** Add a default cap (e.g., `take: 500`) or implement cursor-based pagination on this endpoint.
**Severity:** P2 — performance risk at scale; no immediate exploit.

### M2 — `buyerMatchService` Loads All Users on Every Sale Publish
**File:** `packages/backend/src/services/buyerMatchService.ts` line 140
**Route:** Triggered non-blocking on `PATCH /api/sales/:id` (sale publish)
**Finding:** `notifyMatchedBuyers()` calls `prisma.user.findMany({ where: { role: 'USER' } })` with no limit, then scores each user individually with additional DB queries. This is fire-and-forget so it won't block the publish response, but as the user base grows it will create sustained DB load on every sale publish.
**Fix:** Add `take: 1000` limit and/or implement a pre-filtered interest-based query to reduce the candidate set before scoring.
**Severity:** P2 — scale risk; acceptable for current beta volume.

### M3 — Snooze Reactivation Service Is a No-Op Placeholder
**File:** `packages/backend/src/services/snoozeService.ts` line 96
**Finding:** `checkAndReactivateSnoozes()` has a TODO comment and logs completion without actually querying for expired snoozed subscribers. The snooze audit table is not yet implemented. If this service is called on a schedule, it silently does nothing.
**Fix:** Either implement the audit table and query, or document explicitly that snooze reactivation is not yet active. Remove the misleading "check complete" log.
**Severity:** P2 — silent feature gap; could mislead Patrick into thinking snoozed subscribers are being reactivated.

### M4 — Stale Comment in `.env.example` References Decommissioned Neon
**File:** `packages/backend/.env.example` line 11
**Finding:** The comment reads `# Direct connection for migrations (Neon)`. Neon was decommissioned in S264; the project now uses Railway PostgreSQL. The actual env values are correct (confirmed Railway hostname), but the comment will confuse future contributors or onboarding.
**Fix:** Update the comment to `# Direct connection for migrations (Railway)`.
**Severity:** P3 — documentation drift; no functional impact.

## High (fix this week)

**H1 — Planner AI endpoint is unauthenticated (AI cost abuse risk)**
File: `packages/backend/src/routes/planner.ts`
The `POST /api/planner/chat` endpoint is intentionally public (no `authenticate` middleware). It calls Anthropic API and deducts from the AI cost ceiling. While a rate limiter (30 req / 15 min per IP) and cost ceiling check are in place, IP-based limits are easily bypassed with rotating IPs/proxies. An attacker can systematically exhaust the AI cost ceiling, effectively DoS-ing the feature for real users. Recommend adding `authenticate` middleware so only logged-in users can access the planner, or at minimum tying rate limits to user ID rather than IP.

**H2 — `getSaleBounties` leaks submitter email to all authenticated users**
File: `packages/backend/src/controllers/bountyController.ts` line ~108
The `getSaleBounties` endpoint (which shows bounties attached to a specific sale) returns `user: { select: { name: true, email: true } }`. Any authenticated user who knows a `saleId` can retrieve the email addresses of all bounty submitters for that sale. Submitter name is fine; email is PII and should be excluded from this response.

## Medium (fix this sprint)

**M1 — Multiple `findMany` queries without `take` limits (performance time bomb)**
Files: `bountyController.ts`, `arrivalController.ts`, `betaInviteController.ts`, `boostController.ts`, `brandFollowController.ts`, `buyingPoolController.ts`, `couponController.ts`, `crewController.ts`, `csvExportController.ts`, `disputeController.ts`, and more.
As data grows these queries will pull unbounded result sets. Admin-only controllers (`adminController.ts`) correctly paginate; the pattern needs to propagate to user-facing controllers. Recommend adding `take: 100` (or appropriate per-entity limit) as a default. Export/CSV endpoints are an intentional exception.

**M2 — `include: { user: true }` / `include: { organizer: true }` without select scope**
Files: `billingController.ts` (line 35), `fraudController.ts` (lines 168, 224), `checklistController.ts` (lines 58, 105, 181, 245).
These patterns load the full related record including any fields added to the schema over time (e.g., internal flags, tier info, sensitive fields). They are used for internal authorization checks (not returned directly to clients in most cases), but it's safer to use `select` to only pull the fields actually needed. The checklist controller's `include: { organizer: true }` is used only to check `sale.organizer.userId` — it should be `select: { userId: true }`.

**M3 — `bountyController.ts` Stripe integration incomplete (checkoutUrl: null)**
File: `packages/backend/src/controllers/bountyController.ts` line 503
`checkoutUrl: null` with a `// TODO: integrate Stripe` comment is returned to clients in the bounty purchase flow. If any frontend code tries to redirect to `checkoutUrl`, it will silently fail. This is a functional gap in an active user flow.

**M4 — fraudService auto-suspension is unimplemented**
File: `packages/backend/src/services/fraudService.ts` line 94
The fraud service detects high-risk bids but has `// TODO: Set suspendedAt on User once field added to schema (#73-phase3)`. Fraud signals are logged but automatic suspension is not executing, meaning the system flags fraudsters without acting.

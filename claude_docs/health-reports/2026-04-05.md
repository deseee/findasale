### M2 — `buyerMatchService` Still Loads All Users Without Limit *(carried from 2026-03-29)*
**File:** `packages/backend/src/services/buyerMatchService.ts` line 140
**Finding:** `prisma.user.findMany({ where: { role: 'USER' } })` has no `take` limit. Loads every shopper in the database, then scores each one individually with additional DB queries. This runs as a background fire-and-forget on every sale publish. As the user base grows this will cause escalating DB load on every publish event.
**Fix:** Add `take: 1000` to the query (matches the practical notification cap). Optionally add a pre-filter using `categoryInterests` to reduce the candidate set before scoring.
**Severity:** P2 — scale risk; not blocking at current beta volume but will worsen linearly with user growth.

---

### M3 — Snooze Reactivation Is a Silent No-Op *(carried from 2026-03-29)*
**File:** `packages/backend/src/services/snoozeService.ts` line 95
**Finding:** `checkAndReactivateSnoozes()` logs `"check complete"` but does not actually query for expired snoozes — the audit table does not exist yet. If this service is called on a schedule it silently pretends to work.
**Fix:** Either implement the `snoozeAudit` table and query, or replace the misleading success log with `console.warn('[snooze] Reactivation check: not yet implemented')` so the gap is surfaced in Railway logs.
**Severity:** P2 — silent feature gap; snoozed subscribers may never be reactivated.

### M5 — Export Controllers Fetch All Rows Without a Cap *(new)*
**Files:**
- `packages/backend/src/controllers/csvExportController.ts` line 66 — `prisma.item.findMany({ where: { saleId } })` with no `take`. Returns all items in a sale.
- `packages/backend/src/controllers/earningsPdfController.ts` line 22 — `prisma.sale.findMany({ where: { organizerId } })` with no `take`. Returns all sales + all purchases for a full year.

**Finding:** For export endpoints, fetching all rows is intentional (the user wants their full dataset). But with no cap, a large sale (1,000+ items) or a high-volume organizer's annual earnings PDF will pull a very large payload in a single synchronous query, likely causing a 30-second Railway timeout and OOM.
**Fix (short-term):** Add a `take: 2000` safety cap and return a warning header if the cap is hit. Long-term: move to streaming or chunked export.
**Severity:** P2 — will silently time out for any organizer with a large catalog. Not critical today, but a real user-facing failure waiting to happen.

---

### M6 — Snooze Controller Logs Subscriber Email Addresses *(new)*
**File:** `packages/backend/src/controllers/snoozeController.ts` lines 29, 58, 84
**Finding:** Three `console.log` calls include a subscriber's email address directly:
```
[snooze] Webhook: unsubscribe from user@example.com
[snooze] Status check for user@example.com
[snooze] Manual reactivation for user@example.com
```
These end up in Railway production logs. Email addresses are PII. Any log aggregation tool, Railway team member with log access, or future log export would expose subscriber emails.
**Fix:** Replace email with an anonymized identifier (e.g., first 3 chars + `***` + domain, or a hash). Example: `user@example.com` → `use***@example.com`.
**Severity:** P2 — minor PII leak via logs; low exploit risk but a compliance and trust concern.

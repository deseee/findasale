## Database Layer — Schema is SOURCE OF TRUTH

### Prisma Schema
- **Location:** `packages/database/prisma/schema.prisma` (1936 lines)
- **Authority:** This is THE SOURCE OF TRUTH for all model field names, relations, and constraints
- **How to use:**
  1. Before writing ANY code that references a model field, read the schema
  2. Search for the model name (e.g., `model User {`)
  3. Verify the field name, type, and relations
  4. If a field doesn't exist in the schema, it doesn't exist in the database

### Generated TypeScript Types
- **Client location:** `packages/database/node_modules/.prisma/client/index.d.ts` (auto-generated by `prisma generate`)
- **Re-exported from:** `packages/database/src/index.ts` (all Prisma types exported here for other packages)
- **How to use:** Backend imports types via:
  ```typescript
  import { User, Sale, Item, Organizer, Notification } from '@findasale/database';
  ```
- **Frontend:** Do NOT import from `@findasale/database` directly (Vercel build fails). Instead:
  - Infer types from backend API responses
  - Use context providers (AuthContext) for auth types
  - Define local interfaces for component props

### Migrations
- **Location:** `packages/database/prisma/migrations/` (numbered folders: `YYYYMMDDHHMMSS_migration_name/`)
- **Each migration contains:**
  - `migration.sql` — actual SQL executed against Neon
  - `migration_lock.toml` — lockfile (commit this)
- **When to add:** `pnpm exec prisma migrate dev --name feature_name` (generates SQL)
- **When to deploy:** Manual step before shipping code:
  ```bash
  cd packages/database
  npx prisma migrate deploy   # against Neon (override DATABASE_URL)
  npx prisma generate         # regenerate TS client
  ```

## Backend Layer — Controllers Own Contracts

### Controllers (Business Logic & Response Shapes)
- **Location:** `packages/backend/src/controllers/` (96 files)
- **Naming:** `[domain]Controller.ts` (e.g., `saleController.ts`, `itemController.ts`, `billingController.ts`)
- **What each controller has:**
  - `export const getX = async (req, res) => { ... }` (GET handler)
  - `export const createX = async (req, res) => { ... }` (POST handler)
  - `export const updateX = async (req, res) => { ... }` (PUT/PATCH handler)
  - `export const deleteX = async (req, res) => { ... }` (DELETE handler)
- **Response contract (ALL controllers):**
  ```typescript
  res.json({
    success: boolean,
    data?: T,  // The actual response (shape defined inline in controller)
    message?: string
  })
  ```
- **Error contract:**
  ```typescript
  res.status(4xx or 5xx).json({
    message: "Human-readable error",
    code?: "OPTIONAL_ERROR_CODE"
  })
  ```
- **How to use:**
  1. Identify the endpoint you need (e.g., `/api/sales/{id}`)
  2. Find the corresponding controller (e.g., `saleController.ts`)
  3. Read the handler function's response type (check `res.json({ ... })`)
  4. That shape is YOUR API contract — don't assume

### Routes (Thin Routing Layer)
- **Location:** `packages/backend/src/routes/` (38 files)
- **Naming:** `[domain].ts` (e.g., `sales.ts`, `items.ts`)
- **What they do:** Import controllers, define Express routes, apply middleware
- **Example:**
  ```typescript
  import { getSaleById, updateSale } from '../controllers/saleController';
  import { requireOrganizer } from '../middleware/auth';

  router.get('/:id', getSaleById);
  router.put('/:id', requireOrganizer, updateSale);
  ```
- **How to find the right route:**
  1. Take the endpoint path (e.g., `/api/sales/:id`)
  2. Remove `/api/` prefix → `sales/:id`
  3. Find the domain (first part) → `sales`
  4. Open `packages/backend/src/routes/sales.ts`
  5. Find the matching method and path
  6. See which controller function is called
  7. Read the controller to understand the contract

### Middleware (Auth, Rate Limiting, Timeouts)
- **All middleware location:** `packages/backend/src/middleware/`
- **Auth middleware:** `auth.ts`
  - Validates JWT from Authorization header
  - Attaches `req.user` (User object with `id`, `email`, `roles`, `subscriptionTier`)
  - `requireOrganizer` gates to `req.user.roles.includes('ORGANIZER') || req.user.role === 'ADMIN'`
  - `requireAdmin` gates to `req.user.role === 'ADMIN'`
- **Rate limiters:** All in `packages/backend/src/index.ts`
  - `globalLimiter` (15 req/min per IP, Redis-backed)
  - `authLimiter` (5 auth attempts/min per IP, Redis-backed)
  - `fileUploadLimiter` (10 uploads/min per user, Redis-backed)
  - `webhookLimiter` (100 requests/min per organizer, in-memory)
- **Request timeout:** `requestTimeout.ts`
  - 30s timeout on all requests
  - Returns 503 if exceeded

## Frontend Layer — Hooks & API Client

### API Client (Axios Instance)
- **Location:** `packages/frontend/lib/api.ts`
- **What it does:**
  - Base URL: `process.env.NEXT_PUBLIC_API_URL || '/api'`
  - Auto-includes JWT from `localStorage.token` in Authorization header
  - Auto-includes CSRF token in `x-csrf-token` header for POST/PUT/PATCH/DELETE
  - Handles 401 redirects to `/login`
  - Surfaces Zod validation errors in response
- **How to use:**
  ```typescript
  const response = await api.get('/sales/:id');
  const data = response.data;  // API response (with { success, data, message } structure)
  ```
- **All API calls go through this instance** — no direct axios calls

### Hooks (Data Fetching & Real-Time)
- **Location:** `packages/frontend/hooks/` (51 custom hooks)
- **CRITICAL RULE:** Hook return shapes are INCONSISTENT. Always read the hook file before destructuring.
- **Pattern examples:**
  - React-query hooks return `{ data, isLoading, isError, error }` object
  - Some hooks return `[data, loading, error]` array
  - Some hooks return custom objects like `{ isFavorited, isLoading }`
- **How to find the right hook:**
  1. Identify the data you need (e.g., "get current user's subscription")
  2. Search `packages/frontend/hooks/` for a matching hook (e.g., `useSubscription.ts`)
  3. Open it and read the `return` statement to see the exact shape
  4. Use that shape when destructuring in your component
- **Hooks list (sample):**
  - `useSubscription()` → subscription tier, status, renewal date
  - `useUsageStats()` → sales count, items count, photos count, tier limits
  - `useFavorite(itemId)` → toggle favorite, get favorite status
  - `useConversations()` → list of buyer-organizer conversations
  - `useSendMessage(conversationId)` → send message to conversation
  - `useRipples()` → real-time engagement notifications
  - `useReputation(organizerId)` → organizer reputation score
  - `useCommandCenter()` → multi-sale dashboard metrics
  - `usePasskey()` → register/authenticate via WebAuthn
  - See `packages/frontend/hooks/` directory for full list

### Auth Context
- **Location:** `packages/frontend/context/AuthContext.tsx`
- **What it holds:**
  - Current user (parsed JWT)
  - User roles array
  - Subscription tier
  - Login/logout functions
- **How to use:**
  ```typescript
  import { useAuth } from '../context/AuthContext';

  const { user, logout } = useAuth();
  console.log(user.roles);  // ["USER", "ORGANIZER"]
  console.log(user.subscriptionTier);  // "PRO"
  ```

## Authentication Contract

### JWT Payload
When decoded, the JWT contains:
```typescript
{
  id: string;                                         // user.id
  email: string;                                      // user.email
  name: string;                                       // user.name
  role: string;                                       // DEPRECATED — use roles array
  roles: string[];                                    // ["USER"] or ["USER", "ORGANIZER"] or ["USER", "ORGANIZER", "ADMIN"]
  subscriptionTier: "FREE" | "SIMPLE" | "PRO" | "TEAMS";
  iat: number;                                        // issued at (Unix timestamp)
  exp: number;                                        // expires at (Unix timestamp)
}
```

### Flow
1. User logs in via `/api/auth/login` or `/api/auth/passkey-authenticate`
2. Backend returns JWT in response body (or cookie, if configured)
3. Frontend stores JWT in `localStorage.token`
4. On every API call, `packages/frontend/lib/api.ts` auto-includes JWT in Authorization header
5. Backend middleware validates JWT and attaches `req.user`
6. If JWT is invalid/expired, API returns 401; frontend redirects to `/login`

### Roles & Permissions
- **USER (Shopper):** Can browse items, favorite, bid, purchase, leave reviews, subscribe to sales
- **ORGANIZER:** Can create/manage sales, list items, view analytics, manage team, invoice shoppers
- **ADMIN:** Can moderate content, manage organizers, view all sales, change tiers, send announcements
- **Dual-role (Feature #72):** A user can have `roles: ["USER", "ORGANIZER"]` and hold separate subscriptions per role

## Real-Time Contracts (WebSocket)

### Server Setup
- **Transport:** Socket.io on backend
- **URL:** `process.env.NEXT_PUBLIC_SOCKET_URL || 'http://localhost:3001'`
- **Singleton:** One socket per frontend session

### Rooms & Events
- **sale:{saleId}** → Sale status, item sales, bid updates
  - Events: `SALE_STATUS_UPDATE`, `ITEM_SOLD`, `BID_PLACED`
- **user:{userId}** → Personal notifications
  - Events: `MESSAGE_RECEIVED`, `TIER_UPDATED`, `SALE_REMINDER`, `BADGE_UNLOCKED`

### How to subscribe
```typescript
import { io } from 'socket.io-client';

const socket = io('http://localhost:3001');
socket.emit('join_room', { saleId: '123', userId: 'user-456' });
socket.on('ITEM_SOLD', (data) => {
  console.log('Item sold:', data);
});
```

## Error Codes & Status Codes

### HTTP Status Codes (Backend)
| Code | Meaning | Response Body |
|------|---------|---------------|
| 200 | Success | `{ success: true, data: T }` |
| 400 | Validation error (Zod) | `{ message: "...", code: "VALIDATION_ERROR" }` |
| 401 | Unauthorized (invalid/missing JWT) | `{ message: "Unauthorized" }` |
| 403 | Forbidden (role/permission gate) | `{ message: "You must be an organizer", code: "FORBIDDEN_ROLE" }` |
| 404 | Resource not found | `{ message: "Sale not found" }` |
| 429 | Rate limit exceeded | `{ message: "Too many requests", code: "RATE_LIMIT" }` |
| 500 | Server error (logs to Sentry) | `{ message: "Internal server error" }` |
| 503 | Service unavailable (external API down) | `{ message: "Service temporarily unavailable" }` |

## Performance & Caching

### Query Stale Times (react-query)
- Subscription tier: 5 minutes
- Usage stats: 5 minutes
- Sale details: 30 seconds (fresh for real-time changes)
- Item list: 30 seconds
- User profile: 1 minute
- Conversations: 10 seconds (real-time via WebSocket)

### Cache Busting
- On successful mutation (POST/PUT/PATCH), invalidate related query keys
- Example: After creating an item, invalidate `['sales', saleId, 'items']` query
- Use react-query's `queryClient.invalidateQueries()`

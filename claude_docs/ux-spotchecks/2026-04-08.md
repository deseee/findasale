### Issues Found

#### Line Queue — `[id].tsx`

- **[HIGH] React Rules of Hooks violation** — `line-queue/[id].tsx:54–69` — A conditional `router.push('/login'); return null` at lines 54–57 appears before the `useCallback` hook definition at line 59. React requires hooks to be called in the same order every render — placing a hook after a conditional return is a violation that can cause runtime errors and warnings in strict mode. The `useCallback` must be moved above the auth guard or the auth guard must be restructured as an effect. — **Severity: high**

- **[MEDIUM] Silent API failure hides real errors** — `line-queue/[id].tsx:64–65` — The `loadStatus` catch block silently swallows all errors with a comment "Silently ignore poll errors." If the API endpoint `/lines/${saleId}/status` fails on the initial load, `isLoading` is set to false in `finally` and the user sees the empty state ("No one in line yet") rather than an error. A shopper line could be completely full and the organizer would have no indication that data failed to load. At minimum, the first-load error should surface a toast or inline message; subsequent poll errors can remain silent. — **Severity: medium**

- **[LOW] Loading state is plain text, not skeleton** — `line-queue/[id].tsx:160–162` — While loading, the page shows "Loading line…" plain text. All other pages in the app use `<Skeleton>` components. Low cosmetic inconsistency. — **Severity: low**

#### Shopper Dashboard — `dashboard.tsx`

- **[HIGH] Pickups tab has no content** — `dashboard.tsx` — The tab bar includes a "Pickups" tab (`{ id: 'pickups', label: 'Pickups' }`), and `MyPickupAppointments` is imported at line 28, but there is no `{activeTab === 'pickups' && ...}` content block in the file. Clicking the Pickups tab renders a completely blank area below the tab bar. UX dead end. `MyPickupAppointments` needs to be wired into a Pickups tab content block. — **Severity: high**

- **[MEDIUM] `alert()` used instead of toast for referral copy** — `dashboard.tsx:140` — `navigator.clipboard.writeText(...).then(() => { alert('Referral link copied to clipboard!'); })` uses a native browser alert dialog. Every other action in the app uses `showToast`. The native alert is jarring, blocks the UI, and is inconsistent with brand design. Replace with `showToast('Referral link copied!', 'success')`. — **Severity: medium**

- **[MEDIUM] "You have items saved" shown unconditionally to all returning shoppers** — `dashboard.tsx:320–333` — The returning-shopper hero section shows a blue banner reading "You have items saved. Ready to continue your hunt?" for every returning shopper who has no pending invoices. The `hasSavedItems` variable (line 264) is always `false` and is never used in the condition. The banner fires regardless of whether the user has any items saved. A returning shopper with zero saved items will see a false "items saved" message every visit. Either wire `hasSavedItems` to real data or suppress this banner until the collections API is connected. — **Severity: medium**

- **[LOW] `console.error` in production page** — `dashboard.tsx:99, 115` — Two `console.error` calls for failed referral code and QR code fetches. These won't affect behavior but appear in browser dev tools for all users. Should be removed or replaced with silent failure. — **Severity: low**

- **[LOW] `purchase.purchasedAt` rendered as raw string without formatting** — `dashboard.tsx:777` — Purchase date is output directly as `{purchase.purchasedAt}` with no date formatting. Depending on what the API returns (likely an ISO timestamp), users may see `2026-03-14T18:22:41.000Z` instead of a readable date. Should use `new Date(purchase.purchasedAt).toLocaleDateString()`. — **Severity: low**

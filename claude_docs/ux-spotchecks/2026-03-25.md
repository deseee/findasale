# UX Spot-Check — 2026-03-25
## Flow: Organizer Dashboard + Edit Sale
## Files Reviewed:
- `packages/frontend/pages/organizer/dashboard.tsx`
- `packages/frontend/pages/organizer/edit-sale/[id].tsx`

---

### Issues Found

- **No error state for failed sale fetch in Edit Sale** — `edit-sale/[id].tsx:70-77` — **severity: high**
  The `useQuery` for `/sales/${id}` does not destructure or handle `isError`. If the API returns a 404, 500, or network error, the form renders silently with all blank/default field values. The organizer has no indication that something went wrong — they see an empty form with no feedback. Fix: add `isError` check and render an error message with a retry option (or redirect to dashboard).

- **No error states for any of 6 data queries in Dashboard** — `dashboard.tsx:91-243` — **severity: medium**
  `salesData`, `analyticsData`, `orgProfile`, `tierData`, `earnings`, and `holdCountData` queries all omit `isError` handling. If any request fails: stats cards silently show `0`, tier/reputation cards simply don't render (conditional on data presence), and sales tab returns empty state with no failure signal. Users have no way to distinguish "no data" from "fetch failed." Fix: at minimum, add a top-level error banner when `salesData` or `orgProfile` fail, as those are the two most critical data dependencies on this page.

- **Sales tab loading state is plain text, not skeleton** — `dashboard.tsx:838` — **severity: medium**
  The rest of the page uses `<Skeleton />` components for loading states (auth load at line 254-268 renders 3 skeleton cards). The Sales tab loading path renders `<p>Loading your sales...</p>` — plain unstyled text. This is inconsistent and jarring compared to the rest of the page. Fix: replace with skeleton cards matching the sale card layout.

- **Tab navigation missing ARIA roles** — `dashboard.tsx:641-652` — **severity: medium**
  The Overview/Sales tab buttons are styled as tabs but have no `role="tab"`, `aria-selected`, or wrapping `role="tablist"`. Screen reader users cannot identify these as a tab interface or know which tab is active. Fix: add `role="tablist"` to the wrapper div, `role="tab"` and `aria-selected={activeTab === tab}` to each button.

- **`console.error` in production code** — `dashboard.tsx:212` — **severity: low**
  `console.error('Clone failed:', error)` fires in the `handleCloneSale` error handler. The error is also shown via `showToast` so the console call is redundant. Fix: remove the `console.error` line.

- **Hardcoded "Coming Soon" in rendered UI** — `dashboard.tsx:821-822` — **severity: low**
  The "Plan a Sale" card renders a visible "Coming Soon" badge to all organizers. This is an announced future feature — intentional — but it is hardcoded placeholder content in the production UI per check criterion #7. Acceptable to leave as-is if intentional; flag for awareness.

- **Upgrade CTA dismiss button missing `aria-label`** — `dashboard.tsx:628-629` — **severity: low**
  The "✕" dismiss button on the upgrade CTA uses `title="Dismiss"` (tooltip, not accessible name). Screen readers may read the ✕ character without context. Fix: add `aria-label="Dismiss upgrade prompt"`.

### Clean

- **SSR safety** — All `localStorage`, `window.matchMedia`, and `window.*` accesses in `dashboard.tsx` are correctly wrapped in `useEffect` with `typeof window === 'undefined'` guards. No SSR violations.
- **SSR safety (edit-sale)** — `window.confirm()` and `window.location.reload()` only appear inside event handlers (click, not render), which is SSR-safe.
- **Edit-sale loading state** — Auth and data loading are combined into a single graceful skeleton (lines 217-232) using the `<Skeleton />` component. Clean.
- **Edit-sale error feedback** — `updateMutation`, `handleCloneSale`, `handleToggleSaleStatus`, `handleGenerateDescription`, and `handleAlaCarteSuccess` all have `onError` / `catch` handlers that route to `showToast`. No silent failures on write operations.
- **Form labels (edit-sale)** — All form inputs have associated `<label>` elements. Title, Description, Start Date, End Date, Address, City, State, ZIP, and Neighborhood all properly labeled.
- **Empty states (dashboard)** — Sales tab has a proper `<EmptyState>` component for the zero-sales case with a CTA to create a first sale.
- **No console.log in edit-sale** — Zero console noise in `edit-sale/[id].tsx`.
- **No missing alt text** — No `<img>` tags found in either file; all visual elements use SVG icons or emoji spans.
- **Loading feedback on async buttons** — "Saving...", "Duplicating...", "Updating...", "Generating…", "Cloning..." disabled-state labels are present on all async action buttons in both files.
- **Publish confirmation flow** — Publish/Unpublish toggle uses `window.confirm()` for confirmation before API call. While not a custom modal, it prevents accidental status changes.

2. **(MEDIUM) dashboard.tsx** — Add `isError` to the `salesData` and `orgProfile` queries at minimum. Display a banner like "Some data failed to load — try refreshing" when critical queries fail.

3. **(MEDIUM) dashboard.tsx:838** — Replace `<p>Loading your sales...</p>` with skeleton cards in the sales tab loading branch.

4. **(MEDIUM) dashboard.tsx:641-652** — Add `role="tablist"` to the tab wrapper, `role="tab"` and `aria-selected` to each tab button.

5. **(LOW) dashboard.tsx:212** — Remove `console.error('Clone failed:', error)` line.

6. **(LOW) dashboard.tsx:628-629** — Add `aria-label="Dismiss upgrade prompt"` to the dismiss button.

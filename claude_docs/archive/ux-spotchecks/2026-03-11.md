### Issues Found

**[id].tsx — line-queue**

- **Rules of Hooks violation** — `pages/organizer/line-queue/[id].tsx` lines 55–58 — severity: **high**
  Conditional early return (`if (!authLoading && (!user || ...)) return null`) appears after
  `useState` hooks (lines 51–53) but *before* `useCallback` (line 60) and `useEffect` (line 72).
  React requires hooks to be called in the same order on every render — this violates that rule.
  In practice the bug lies dormant because auth always starts as `isLoading: true`, but React
  DevTools and lint rules will flag it, and it can surface in edge-case server-side render paths.
  Fix: move the auth guard after all hook declarations.

- **Silent initial load failure** — `pages/organizer/line-queue/[id].tsx` line 65 — severity: **medium**
  `loadStatus()` catch block is intentionally silent (`// Silently ignore poll errors`).
  This is fine for background polls, but it also swallows the *initial* load error.
  If the first request fails, `isLoading` is set to `false` (finally block runs) and the page
  shows the empty state ("No one in line yet") with no indication anything went wrong.
  Organizer has no feedback and may assume the queue is genuinely empty.
  Fix: distinguish first-load errors from poll errors and show a toast or inline error on first fail.

- **"Entered ✓" checkmark character** — `pages/organizer/line-queue/[id].tsx` line 200 — severity: **low**
  Raw unicode ✓ in button text may be announced as "check mark" or "tick" by screen readers.
  Fix: wrap in `<span aria-hidden="true">` or use an SVG icon with the label coming from button text.

- **Call Next ignores no-phone entries** — `pages/organizer/line-queue/[id].tsx` lines 137–147 — severity: **low**
  The "Call Next" button triggers an SMS to the front of the queue without checking whether
  `waiting[0]?.user.phone` is null. The row already displays "No phone — cannot SMS" as a
  warning, but the button doesn't disable or warn when the next-up person has no phone.
  Fix: disable the button or show a pre-confirm warning when `waiting[0]?.user.phone === null`.

**dashboard.tsx — shopper**

- **Rules of Hooks violation** — `pages/shopper/dashboard.tsx` lines 37–40 — severity: **high**
  Same pattern as line-queue: conditional early return (`if (!isLoading && !user) return null`)
  after `useAuth` and `useState` but before three `useQuery` calls (lines 42, 51, 60).
  Same risk and fix as above — move auth guard below all hook declarations.

- **No loading/error states for tab queries** — `pages/shopper/dashboard.tsx` lines 42–67 — severity: **medium**
  The `useQuery` calls for `purchases`, `favorites`, and `userData` don't destructure `isLoading`
  or `isError`. In the Purchases tab, if the query is still in-flight, `purchases` is `undefined`,
  so `purchases && purchases.length > 0` is `false` — the empty state ("No purchases yet") flashes
  briefly before data loads. If the query fails, the empty state is shown permanently with no error.
  Fix: destructure `isLoading`/`isError` from each query and render a Skeleton or error message
  inside each tab while data is pending or failed.

- **Subscribed tab is a static stub** — `pages/shopper/dashboard.tsx` lines 212–226 — severity: **medium**
  The "Subscribed" tab always shows an empty state, regardless of whether the user actually has
  subscribed sales. There is no data fetch for subscriptions. If the subscriptions feature is live
  (or partially live), subscribed users see misleading empty content.
  Fix: either fetch subscriptions data and render it, or add a visible "Coming soon" label to the
  tab so users understand it's not functional yet.

- **Tab list missing ARIA roles** — `pages/shopper/dashboard.tsx` lines 94–114 — severity: **low**
  The tab container is a plain `div` with no `role="tablist"`, and individual tab buttons have no
  `role="tab"` or `aria-selected`. Screen readers cannot identify this as a tab interface.
  Fix: add `role="tablist"` to the container div and `role="tab"` + `aria-selected={activeTab === tab.id}`
  to each button.

- **`userData` query is never consumed** — `pages/shopper/dashboard.tsx` lines 60–67 — severity: **low**
  `useQuery` fetches `/users/me` on every dashboard load, but the resulting `userData` is never
  referenced in the JSX. Dead API call on every page load.
  Fix: remove the query unless a downstream component needs it, or pass `userData` to a child
  component that requires it.

- **Raw ISO date string in Purchases tab** — `pages/shopper/dashboard.tsx` line 153 — severity: **low**
  `purchase.purchasedAt` is rendered directly without formatting. Users see strings like
  `2026-03-11T14:22:00.000Z` instead of "March 11, 2026".
  Fix: wrap in `new Date(purchase.purchasedAt).toLocaleDateString()` or a date formatting utility.

## Medium (fix this sprint)

**M1 тАФ message.findMany: no take limit on thread history**
- File: `packages/backend/src/controllers/messageController.ts:86`
- `message.findMany({ where: { conversationId }, ... })` has no `take:` limit.
  At beta scale with low volume this is safe, but a conversation with 10k+ messages
  would return the full set тАФ slow query + large payload.
- Fix: add `take: 200, orderBy: { createdAt: 'desc' }` and add a "load earlier messages"
  pagination pattern when needed.

**M2 тАФ conversation.findMany: no take on inbox list**
- File: `packages/backend/src/controllers/messageController.ts:20,35`
- Organizer and shopper inbox lists have no hard `take:` cap. Scoped to userId so safe
  at beta, but worth adding `take: 100` as a guardrail before real users accumulate conversations.
- Fix: add `take: 100` to both inbox queries.

**M3 тАФ contact route: only covered by global rate limiter (200/15min)**
- File: `packages/backend/src/index.ts:164`
- `/api/contact` POST is a public email-sending endpoint covered only by the global
  200-req/15min limiter. A single bad actor could send ~200 spam emails per 15 minutes.
- Fix: Create a dedicated `contactLimiter` (e.g., 5 req/15min) and apply it to
  `/api/contact` similar to how `authLimiter` is applied to `/api/auth`.
